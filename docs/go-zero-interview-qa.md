# go-zero + IM 项目面试问答整理

这份文档基于当前项目背景整理，目标不是背定义，而是让你能把 go-zero、gRPC、Etcd、Redis、Kafka 和 IM 链路结合起来讲清楚。

项目背景默认如下：

- 基于 Go 的 IM/长连接后端
- 技术栈：Go、Go-Zero、gRPC、Etcd、Redis、MySQL、Kafka、Docker、WebSocket
- Gateway 多实例承接 WebSocket 长连接
- 服务间调用使用 go-zero 的 RPC/gRPC
- 用 etcd 做服务注册发现
- 做过跨节点消息路由、在线状态维护、zRPC `p2c_ewma` 负载均衡、群聊异步扩散
- 更像“多实例接入层 + 部分服务拆分”，不是完全拆到底的全微服务

---

## 一、go-zero 基础定位

### 1. go-zero 是什么？和 gin 有什么区别？

**面试官为什么会问这个**

先判断你是“会用框架”还是“理解框架定位”。

**简洁回答版**

go-zero 不只是 HTTP 框架，它更像一套 Go 的服务端工程化框架，既有 API 服务也有 RPC 服务，还有服务发现、限流、负载保护、代码生成这些能力；gin 更偏轻量 HTTP 路由框架。

**展开回答版**

gin 解决的是“怎么写 HTTP 服务”，go-zero 解决的是“怎么把 API、RPC、配置、依赖、治理能力按统一方式组织起来”。很多团队不是不会写 HTTP，而是项目一大就开始乱，handler 很重、依赖到处 new、服务调用散、治理能力靠手搓；go-zero 就是把这些通用问题前置了。

**结合我项目怎么回答**

我项目里 Gateway 用的是 go-zero API，Logic 用的是 go-zero zRPC，WebSocket 也是挂在 Gateway 这层自己补进去的。

**容易踩坑的错误回答**

不要说“go-zero 就是比 gin 快”或者“gin 不能做微服务”，重点不在性能吹法，而在工程能力和边界。

### 2. go-zero 解决的核心问题是什么？

**面试官为什么会问这个**

看你是不是只记住了“框架名字”。

**简洁回答版**

它主要解决的是 Go 服务端工程化落地问题，包括代码骨架、API/RPC 分层、服务发现、配置管理、基础治理和统一开发规范。

**展开回答版**

很多项目不是不会写接口，而是写着写着就开始乱：handler 很重、依赖散、服务调用风格不统一、配置和日志到处都是。go-zero 的价值是先把这些基础盘铺好。

**结合我项目怎么回答**

我项目里最直观的价值就是 Gateway 和 Logic 两类服务都能按统一方式组织，服务入口、依赖注入和中间件都比较规整。

**容易踩坑的错误回答**

不要回答成“解决高并发问题”，高并发不是框架自动解决的，框架只是提供更好的组织和治理基础。

### 3. go-zero 为什么更适合微服务，不只是普通 web 框架？

**面试官为什么会问这个**

看你是否理解“微服务工程能力”。

**简洁回答版**

因为它天然就把 API 服务、RPC 服务、服务发现和治理能力串在一起了，不只是帮你起一个 HTTP server。

**展开回答版**

微服务里要处理的不只是接口路由，还要处理服务拆分、RPC、注册发现、限流、负载保护、统一配置这些问题，go-zero 对这些有原生支持。

**结合我项目怎么回答**

我项目里 Gateway 到 Logic 的内部调用就是 go-zero 的 RPC 思路，Gateway 对外是 API，对内走 RPC，这种组合很适合服务拆分后的边界。

**容易踩坑的错误回答**

不要把“用了 zrpc”直接等于“完整微服务”，微服务是架构方式，不是引个库就算。

### 4. go-zero 的 API 和 RPC 分别是什么？你项目里分别怎么用？

**面试官为什么会问这个**

这是 go-zero 最基础的分层问题。

**简洁回答版**

API 主要面向客户端请求，RPC 主要面向服务间调用；我项目里 Gateway 用 API 承接 HTTP/WebSocket 接入，Logic 用 RPC 承接内部业务调用。

**展开回答版**

API 层更适合登录、历史消息查询、WebSocket 握手这类客户端访问；RPC 层更适合 Gateway 调 Logic 做登录校验、消息编排、历史查询这些内部调用。

**结合我项目怎么回答**

`cmd/gateway` 是 API 服务，`cmd/logic` 是 RPC 服务，WebSocket 在 Gateway 上，消息业务处理在 Logic 上。

**容易踩坑的错误回答**

不要说“API 就是外网，RPC 就是内网”这么绝对，本质是职责划分，不是网络边界定义。

### 5. 你项目里到底是只用了 go-zero 的 API，还是也用了 RPC？

**面试官为什么会问这个**

判断你是不是只会用 API 脚手架。

**简洁回答版**

两者都用了。

**展开回答版**

Gateway 是 go-zero 的 API 服务，Logic 是 go-zero 的 zRPC 服务，Gateway 收到客户端请求以后会通过 gRPC 调 Logic。

**结合我项目怎么回答**

登录、查历史、消息转发都经过 Gateway -> Logic 的 RPC 调用，不只是纯 API 项目。

**容易踩坑的错误回答**

不要只说“也用了 gRPC”，要说清是 go-zero 的 zRPC 服务形态。

### 6. goctl 是干什么的？你项目里怎么用到的？

**面试官为什么会问这个**

很多人知道 go-zero，不知道 goctl。

**简洁回答版**

goctl 是 go-zero 的代码生成工具，用来生成 API、RPC、模型等基础代码骨架。

**展开回答版**

它能帮你生成 `handler/logic/svc/types` 这些目录结构，也能从 proto 生成 RPC 代码，减少样板代码。

**结合我项目怎么回答**

我这个项目的 Gateway、Logic 入口结构就是按 goctl/go-zero 官方脚手架整理出来的，后续 WebSocket、Redis 路由、Kafka 群聊链路是我自己补的。

**容易踩坑的错误回答**

不要说“项目全靠 goctl 生成”，生成的是骨架，核心业务还是自己写的。

### 7. go-zero 的代码生成给你带来了什么好处？有没有什么限制？

**面试官为什么会问这个**

看你是不是只会夸工具，不会讲代价。

**简洁回答版**

好处是快、规范、少样板代码；限制是生成的只是骨架，复杂业务还是得自己设计。

**展开回答版**

它把目录结构、依赖注入、接口入口统一了，团队协作会舒服很多；但像 WebSocket 长连接、Redis 路由、Kafka 异步扩散这种复杂链路，还是要自己手工补。

**结合我项目怎么回答**

我项目里 handler/logic/svc 的标准结构是工具帮了忙，但消息协议、ACK、群聊异步化这些全是业务代码。

**容易踩坑的错误回答**

不要说“用了 goctl 所以开发效率特别高，一切都自动了”，这很容易被追问打穿。

## 二、项目架构与 go-zero 落地

### 8. 你这个项目哪些部分体现了 go-zero？

**面试官为什么会问这个**

防止你只会背框架名字。

**简洁回答版**

主要体现在 Gateway 的 API 层、Logic 的 RPC 层，以及两边统一的 `handler/logic/svc` 或 `server/logic/svc` 分层。

**展开回答版**

Gateway 入口、路由注册、中间件、依赖注入是 go-zero 风格；Logic 的 zRPC 启动、RPC 方法承接、ServiceContext 也是 go-zero 风格。

**结合我项目怎么回答**

`cmd/gateway` 和 `cmd/logic` 这两部分最能体现 go-zero，Transfer 则更偏普通 Kafka worker。

**容易踩坑的错误回答**

不要把 Redis/Kafka/MySQL 这些也算 go-zero 的体现，它们是业务中间件，不是 go-zero 本身。

### 9. 你项目里哪些部分是单体特征，哪些部分是分布式特征？

**面试官为什么会问这个**

看你对自己项目定位是否真实。

**简洁回答版**

单体特征主要是单仓库、共享 Redis/MySQL、服务数量不多；分布式特征主要是多 Gateway、多服务通信、Etcd 发现、跨节点消息路由和 Kafka 异步扩散。

**展开回答版**

它不是那种拆成十几个领域微服务的系统，但已经不是单机闭环应用了。

**结合我项目怎么回答**

Gateway 多实例、Logic RPC、Transfer 异步消费、Redis route + Pub/Sub、Etcd 发现都属于分布式协作。

**容易踩坑的错误回答**

不要硬说“完全微服务”，你这个项目更准确是“多实例接入层 + 部分服务拆分”。

### 10. 多个 Gateway 实例为什么不等于完整分布式？

**面试官为什么会问这个**

看你有没有概念辨析能力。

**简洁回答版**

因为多个 Gateway 只是接入层水平扩容，不代表业务、状态、存储、异步链路都完整分布式了。

**展开回答版**

完整分布式一般还涉及业务服务拆分、状态中心、服务发现、跨节点消息协作、故障隔离等。只有多个 Gateway，更像接入层集群。

**结合我项目怎么回答**

我项目里之所以能说分布式，是因为除了多 Gateway，还有 Logic、Transfer、Redis route、Pub/Sub、Kafka 这些协作链路。

**容易踩坑的错误回答**

不要把“多实例部署”直接等于“完整分布式系统”。

### 11. 你为什么说自己的项目是“多实例部署 + 部分服务拆分”？

**面试官为什么会问这个**

看你是否诚实、有边界感。

**简洁回答版**

因为它确实有多 Gateway、多服务协作，但还没拆到很细的领域微服务。

**展开回答版**

我把接入层、业务层、群聊异步扩散层拆开了，但数据库和 Redis 还是共享，服务数量也不多，所以更像部分服务拆分，而不是全微服务化。

**结合我项目怎么回答**

Gateway、Logic、Transfer 三层已经有明显职责边界，但像用户域、群组域、消息存储域还没继续细拆。

**容易踩坑的错误回答**

不要说“不是完整微服务所以不算分布式”，也不要反过来吹成“大型微服务平台”。

### 12. 你项目里 go-zero 主要是用在网关层、业务层，还是服务通信层？

**面试官为什么会问这个**

看你对框架落点是否清楚。

**简洁回答版**

主要用在 Gateway 接入层和 Logic 业务层，同时也承接了服务间 RPC 通信。

**展开回答版**

Gateway 的 REST/API 结构和 Logic 的 zRPC 结构都用了 go-zero；服务间通信这部分则是 go-zero 的 RPC 思路 + gRPC。

**结合我项目怎么回答**

Transfer 这层不是 go-zero 的重点，更多是普通 Kafka worker。

**容易踩坑的错误回答**

不要答成“全项目都是 go-zero”，你的 WebSocket 管理、Redis/Kafka 逻辑并不是框架直接提供的。

### 13. 如果面试官问“你这个项目到底哪里体现了 go-zero 的价值”，该怎么答？

**面试官为什么会问这个**

这是价值题，不是定义题。

**简洁回答版**

最大的价值是把 Gateway 和 Logic 两类服务用统一规范组织起来，让接入、RPC、依赖和治理能力都更标准。

**展开回答版**

如果不用 go-zero，我也能用 gin + grpc + 自己封装去做，但项目会更碎，目录分层、服务启动、依赖注入、治理能力要自己搭很多。

**结合我项目怎么回答**

我这里最明显的收益是 API/RPC 分层清楚，Gateway 和 Logic 的职责边界更好讲，也更适合后续扩展。

**容易踩坑的错误回答**

不要回答成“它性能更高，所以价值更大”，核心价值不是 benchmark。

### 14. 如果不用 go-zero，你这个项目还能怎么做？为什么最后选 go-zero？

**面试官为什么会问这个**

看你有没有技术选型思维。

**简洁回答版**

能做，可以用 gin + grpc + etcd client + 自己封装中间件来搭；最后选 go-zero，是因为它更省重复建设成本。

**展开回答版**

不用 go-zero 当然能实现，但你要自己搭 API/RPC 骨架、配置管理、服务启动、治理能力和工程规范；go-zero 能把这些基础盘先铺好。

**结合我项目怎么回答**

我这个项目重点在消息路由、ACK、群聊异步化这些业务问题，所以不想把太多精力花在手搓脚手架上。

**容易踩坑的错误回答**

不要说“没有 go-zero 就做不了”，那会显得你过度依赖框架。

## 三、API / RPC / gRPC

### 15. go-zero 的 API 服务和 RPC 服务在职责上怎么划分？

**面试官为什么会问这个**

基础架构题。

**简洁回答版**

API 服务面向客户端接入，RPC 服务面向内部业务调用。

**展开回答版**

API 层更靠近用户和协议，负责请求接入、鉴权、参数整理；RPC 层更靠近业务核心，负责消息编排、顺序控制、落库和异步任务发布。

**结合我项目怎么回答**

Gateway 是 API，Logic 是 RPC。

**容易踩坑的错误回答**

不要把 API 理解成“只能 HTTP”，这里 WebSocket 也是挂在接入层。

### 16. API 层为什么不能把所有业务都干掉？

**面试官为什么会问这个**

看你是否理解分层意义。

**简洁回答版**

因为那样会把接入层做成大而重的业务层，耦合连接管理、协议处理和核心消息逻辑。

**展开回答版**

API 层如果又管连接、又管顺序号、又管群聊扩散、又直接打数据库，会很难扩容，也不利于逻辑复用和服务边界清晰。

**结合我项目怎么回答**

我把 `seq`、路由决策、单聊/群聊分流都放在 Logic，而不是放在 Gateway。

**容易踩坑的错误回答**

不要说“API 层不能做业务”，API 层可以做轻编排，只是不适合承载核心复杂业务。

### 17. 你为什么要做 RPC 拆分？

**面试官为什么会问这个**

这是你项目架构的核心。

**简洁回答版**

因为接入层和消息业务层关注点完全不同，拆开之后更容易扩展和稳定。

**展开回答版**

Gateway 主要关心连接、心跳、鉴权、限流；Logic 主要关心消息编排、路由、顺序和持久化。拆开后可以分别扩容和演进。

**结合我项目怎么回答**

消息是 WebSocket 接进来的，但 `PushMessage` 最终在 Logic 侧做统一处理。

**容易踩坑的错误回答**

不要只说“为了微服务”，太空；要落到职责分离和扩展性。

### 18. go-zero 的 RPC 底层是不是 gRPC？

**面试官为什么会问这个**

看你对 zRPC 的本质理解。

**简洁回答版**

是，go-zero 的 zRPC 底层就是 gRPC。

**展开回答版**

它在 gRPC 之上又帮你做好了配置、服务发现、服务启动、调用封装这些工程能力。

**结合我项目怎么回答**

我项目里 proto 定义、pb 代码、Gateway 调 Logic 的 RPC 本质上都是 gRPC。

**容易踩坑的错误回答**

不要说“zRPC 是 go-zero 自己的一套协议”，不是。

### 19. gRPC 相比 HTTP 调用的优势是什么？

**面试官为什么会问这个**

内部通信选型题。

**简洁回答版**

更适合内部服务调用，协议更紧凑、契约更清晰、生成代码更方便。

**展开回答版**

它有 protobuf 二进制序列化、强接口定义、性能更高、客户端和服务端代码生成方便，这些都适合服务间高频调用。

**结合我项目怎么回答**

Gateway 到 Logic 这条链路用 gRPC，比每次都走 JSON HTTP 更合适。

**容易踩坑的错误回答**

不要把 gRPC 吹成万能，外部开放接口还是 HTTP 更通用。

### 20. 在你的项目里，哪些调用适合 RPC，哪些适合 HTTP？

**面试官为什么会问这个**

看你有没有按场景选协议。

**简洁回答版**

客户端外部访问适合 HTTP/WebSocket，服务内部调用适合 RPC。

**展开回答版**

登录、历史查询、群创建这类客户端请求走 HTTP；长连接消息走 WebSocket；Gateway 到 Logic 的登录校验、推消息、查历史走 RPC。

**结合我项目怎么回答**

这三层在项目里是明确分开的。

**容易踩坑的错误回答**

不要说“HTTP 都可以替换成 RPC”，要考虑客户端兼容性和协议定位。

### 21. 你项目里 WebSocket、HTTP、RPC 三者分别在什么位置？

**面试官为什么会问这个**

综合链路题。

**简洁回答版**

HTTP 负责接入型接口，WebSocket 负责长连接消息通道，RPC 负责 Gateway 和 Logic 之间的内部业务调用。

**展开回答版**

HTTP 是 `/login`、`/history` 这类接口；WebSocket 是 `/ws` 长连接；RPC 是 Gateway 把收到的消息转给 Logic 去做编排。

**结合我项目怎么回答**

这三者在 Gateway/API、Logic/RPC 两层里都有明确位置。

**容易踩坑的错误回答**

不要把 WebSocket 当成 RPC，也不要把 RPC 说成替代 WebSocket。

### 22. 如果面试官追问 proto 文件、序列化、接口定义，应该怎么说？

**面试官为什么会问这个**

验证你是不是只知道“用了 protobuf”。

**简洁回答版**

我项目里用 proto 同时定义了消息结构和 RPC 接口，消息主体统一用 `WireMessage` 这类结构，序列化走 protobuf。

**展开回答版**

客户端消息、心跳、ACK 和内部 RPC 使用的是同一套核心消息模型，这样可以减少模型转换和字段漂移。

**结合我项目怎么回答**

Gateway 收到 WebSocket 二进制帧后先解 protobuf，Logic 内部处理也是围绕这套结构。

**容易踩坑的错误回答**

不要说“Kafka 和 Redis 里全都直接是 protobuf”，项目里 Redis Pub/Sub 外层还有 JSON envelope，Kafka 里是 groupDispatchJob。

## 四、服务注册发现与 etcd

### 23. etcd 在 go-zero 项目里是干什么的？

**面试官为什么会问这个**

高频基础题。

**简洁回答版**

主要做服务注册与发现，让 Gateway 能知道当前有哪些 Logic 节点可用。

**展开回答版**

Logic 节点启动后把地址注册到 etcd，Gateway 再从 etcd 拿到可用节点列表。

**结合我项目怎么回答**

我项目里 etcd 主要服务于 Gateway -> Logic 这一跳。

**容易踩坑的错误回答**

不要把 etcd 说成在线状态中心，那是 Redis route。

### 24. 服务注册发现的流程是什么？

**面试官为什么会问这个**

看你是否真正理解这件事。

**简洁回答版**

服务启动注册到 etcd，调用方从 etcd 取可用节点，再选一个发请求。

**展开回答版**

Logic 通过 zRPC 配置注册自身；Gateway 通过 resolver 获取 `logic` 服务节点列表，再由当前配置的 `p2c_ewma` 选中节点。

**结合我项目怎么回答**

我项目的 `LogicRouter` 就是发现 + 选路 + gRPC client 复用。

**容易踩坑的错误回答**

不要只说“注册到 etcd”就结束，发现和选路要一起说。

### 25. 为什么不能把服务地址写死？

**面试官为什么会问这个**

看你对服务发现意义是否理解。

**简洁回答版**

写死适合单实例，多个 Logic 节点扩缩容、重启、故障摘除时就不灵活了。

**展开回答版**

动态服务发现能让 Gateway 不关心具体实例数量和地址变化。

**结合我项目怎么回答**

我项目有多 Gateway 和可扩展 Logic，如果地址写死，后续扩容很别扭。

**容易踩坑的错误回答**

不要说“写死一定不行”，小项目可以写死，但扩展性差。

### 26. etcd 挂了会怎么样？

**面试官为什么会问这个**

高频追问。

**简洁回答版**

短时间内已有 gRPC 连接和已缓存客户端还能工作，但新节点注册、发现和故障感知会受影响。

**展开回答版**

etcd 是控制面，不是消息主投递链路；它挂了不会瞬间让已建立连接全部失败，但服务发现会退化。生产上一般部署成 etcd 集群。

**结合我项目怎么回答**

我的 Gateway 里对 Logic client 有缓存，所以不是 etcd 一挂全链路立刻断。

**容易踩坑的错误回答**

不要答成“etcd 挂了系统完全没影响”或者“etcd 挂了消息就发不出去了”，都太绝对。

### 27. etcd 和 Nacos 的区别怎么简单回答？

**面试官为什么会问这个**

常见比较题。

**简洁回答版**

两者都能做注册发现，etcd 更偏一致性和基础设施型，Nacos 还把配置中心能力做得更重。

**展开回答版**

面试不用展开太多，知道 etcd 常见于 Go/gRPC 生态，Nacos 在 Java/Spring 生态更常见就够了。

**结合我项目怎么回答**

我项目选 etcd 是因为和 go-zero、gRPC 配合自然。

**容易踩坑的错误回答**

不要现场展开一堆 CAP 八股，项目重点不在这。

### 28. go-zero 为什么常和 etcd 一起用？

**面试官为什么会问这个**

看你是否理解生态组合。

**简洁回答版**

因为 go-zero 的 zRPC 对 etcd 注册发现支持比较自然，服务化落地成本低。

**展开回答版**

框架层已经把服务配置、服务启动、注册发现这些模式统一起来了，所以搭起来很顺。

**结合我项目怎么回答**

我项目里 Logic 侧是 zRPC，Gateway 侧也按 etcd 做 Logic 发现和选路。

**容易踩坑的错误回答**

不要说“go-zero 只能配 etcd”，不是，只是常见组合。

### 29. 你项目中服务启动、注册、发现、调用的大致链路是什么？

**面试官为什么会问这个**

综合链路题。

**简洁回答版**

Logic 启动并注册，Gateway 启动时准备好 LogicRouter，请求进来后按用户 key 选 Logic，再发起 gRPC 调用。

**展开回答版**

Gateway 的 HTTP/WebSocket 接入不直接做核心业务，而是拿 `LogicClient` 去调 Logic；Logic 做完编排后再走 Redis/Kafka 后续链路。

**结合我项目怎么回答**

登录、推消息、查历史基本都是这条链。

**容易踩坑的错误回答**

不要只说“启动后注册到 etcd”，要连到实际调用链上。

### 30. 如果面试官问“你真的理解服务发现吗”，你怎么证明？

**面试官为什么会问这个**

面试官在试你深浅。

**简洁回答版**

我会把注册、租约、发现、选路、缓存客户端这五步完整讲出来。

**展开回答版**

我不仅知道“用 etcd 找服务”，还知道 etcd 解决的是节点列表问题，一致性哈希解决的是稳定选节点，客户端缓存解决的是连接复用。

**结合我项目怎么回答**

我项目里 `LogicRouter` 就体现了发现 + 选路 + client 池，不是只会背概念。

**容易踩坑的错误回答**

不要只回答“我会用 go-zero 配 etcd”，那证明不了理解。

## 五、分布式、高可用、扩容

### 31. 多实例部署的意义是什么？

**面试官为什么会问这个**

基础架构题。

**简洁回答版**

提高吞吐、提升可用性、降低单点风险。

**展开回答版**

多个实例可以分担连接和请求压力，某个节点出故障也不会全站不可用。

**结合我项目怎么回答**

我 10 个 Gateway 的验证，就是接入层多实例意义的直接体现。

**容易踩坑的错误回答**

不要只说“为了高并发”，高可用也是重点。

### 32. 水平扩容和微服务拆分有什么区别？

**面试官为什么会问这个**

概念辨析题。

**简洁回答版**

水平扩容是同一种服务多开几份，微服务拆分是按职责把一个系统拆成多个服务。

**展开回答版**

前者主要解决容量问题，后者主要解决边界和复杂度问题。

**结合我项目怎么回答**

Gateway 多实例是水平扩容，Gateway/Logic/Transfer 拆开是部分服务拆分。

**容易踩坑的错误回答**

不要把两者当同一个概念。

### 33. 你的 Gateway 多实例解决了什么问题？

**面试官为什么会问这个**

项目价值题。

**简洁回答版**

主要解决长连接承载能力、接入层吞吐和单点问题。

**展开回答版**

用户都连在 Gateway 上，多实例之后连接分散到不同节点，跨节点消息则通过 Redis route + Pub/Sub 协作。

**结合我项目怎么回答**

本地 10 Gateway 验证了跨节点路由和多实例接入能力。

**容易踩坑的错误回答**

不要说“解决了一切性能问题”，后面 Redis、Logic 仍可能是瓶颈。

### 34. 跨节点路由为什么是分布式问题？

**面试官为什么会问这个**

看你是否理解“分布式”不是口号。

**简洁回答版**

因为发送方和接收方不在同一台机器上，本机内存状态已经不够用了。

**展开回答版**

你必须有一个共享状态中心知道目标用户在哪台 Gateway，还要有跨节点投递方式把消息送过去。

**结合我项目怎么回答**

我这里共享状态中心是 Redis route，跨节点投递是定向 Pub/Sub。

**容易踩坑的错误回答**

不要把它说成“只是 Redis 发一下而已”，本质是跨节点协作。

### 35. 用户连接落在不同节点时，消息怎么投递？

**面试官为什么会问这个**

这是 IM 项目高频题。

**简洁回答版**

Logic 先查 `route:<uid>` 看目标用户在哪台 Gateway，然后把消息发布到对应 Gateway 专属频道，由目标 Gateway 本机连接表再推给用户。

**展开回答版**

在线就走定向 Pub/Sub，不在线就落 `offline_msg`，同时记录 `pending_ack`。

**结合我项目怎么回答**

这个是单聊主链路核心。

**容易踩坑的错误回答**

不要答成“直接发给对方机器 IP”，项目里是按频道定向，不是直接 TCP 点对点。

### 36. 在线状态为什么要集中维护？

**面试官为什么会问这个**

看你为什么用 Redis。

**简洁回答版**

因为任意 Gateway/Logic 都可能需要知道某个用户当前连在哪台 Gateway。

**展开回答版**

如果在线状态只在本机内存里，跨节点时其他服务就找不到目标用户了。

**结合我项目怎么回答**

我用 Redis 的 `route:<uid>` 维护这张全局在线映射。

**容易踩坑的错误回答**

不要说“为了缓存”，这里本质是共享状态，不是普通缓存。

### 37. 为什么当前使用 p2c_ewma，而不是一致性哈希？

**面试官为什么会问这个**

经常追问。

**简洁回答版**

p2c_ewma 会从候选节点中比较负载估计，优先选择更健康的节点，适合当前无状态 Logic 的负载均衡。若业务确实需要用户粘滞，才考虑一致性哈希。

**展开回答版**

它不是发现服务的工具；Etcd 负责节点发现，p2c_ewma 负责在已发现节点中做负载选择。

**结合我项目怎么回答**

我项目里的 Logic 不保存用户连接状态，因此不要求固定用户命中固定 Logic；Gateway 当前使用 zRPC `p2c_ewma`。代码虽然设置了 hash key，但当前 balancer 不消费该 key，不能据此声称用了哈希路由。

**容易踩坑的错误回答**

不要把 `zrpc.SetHashKey` 当成已经实现一致性哈希；必须同时检查实际 balancer 配置。

### 38. 你的系统是如何避免单点的？

**面试官为什么会问这个**

高可用题。

**简洁回答版**

接入层通过多 Gateway 避免单点，业务层可扩成多 Logic，异步链路用 Kafka/Transfer，状态放 Redis 而不是绑死在单台 Gateway 内存。

**展开回答版**

更严格地说，演示版项目还没有把所有中间件都做到生产级 HA，但服务边界和扩展方向是明确的。

**结合我项目怎么回答**

我会真实说，当前最明确验证的是 Gateway 多实例，Redis/MySQL/Kafka 在生产要进一步做 HA 部署。

**容易踩坑的错误回答**

不要硬说“我项目所有组件都没有单点”，那很容易被追问崩。

### 39. 如果某个 Gateway 挂了，连接和消息会怎么样？

**面试官为什么会问这个**

典型故障题。

**简洁回答版**

这台 Gateway 上的连接会断，用户需要重连；未确认消息会通过 ACK/离线补偿机制兜底。

**展开回答版**

route 会过期或被清理，新消息在目标用户未重新上线前会进离线补偿；重连后按未确认消息重放。

**结合我项目怎么回答**

当前实现里连接不会自动迁移，掉线重连是正常处理方式。

**容易踩坑的错误回答**

不要说“无感切换”，WebSocket 长连接一般做不到完全无感迁移。

### 40. 你项目里有哪些地方体现了分布式协作，而不是单机闭环？

**面试官为什么会问这个**

总括题。

**简洁回答版**

Gateway 到 Logic 的 RPC、Redis 在线状态与 Pub/Sub、Kafka 群聊异步扩散，这三块最能体现。

**展开回答版**

如果只是单机聊天室，这些都不需要。

**结合我项目怎么回答**

跨 Gateway 单聊、群聊异步扩散和 etcd 服务发现是我最典型的分布式协作点。

**容易踩坑的错误回答**

不要只说“用了 Redis/Kafka 就叫分布式”，重点是协作链路。

## 六、go-zero 内置治理能力

### 41. go-zero 里面有哪些内置工程能力？

**面试官为什么会问这个**

看你是不是知道它不只是代码生成。

**简洁回答版**

常见的有配置管理、日志、限流、超时控制、负载保护、服务发现、API/RPC 脚手架这些。

**展开回答版**

不同项目用到的深度不同，不一定全都重度使用。

**结合我项目怎么回答**

我项目里真正落地最明显的是配置、日志、API/RPC 分层和服务发现。

**容易踩坑的错误回答**

不要答太满，好像每个都用得很深。

### 42. 限流、熔断、超时控制、负载保护分别是什么？

**面试官为什么会问这个**

治理基础题。

**简洁回答版**

限流是限制请求速率；熔断是下游异常太多时暂时拒绝调用；超时控制是请求过久自动中断；负载保护是服务压力太大时主动丢弃部分请求保护整体可用性。

**展开回答版**

四者都是为了避免高并发场景下服务整体被拖垮，但触发时机不同。

**结合我项目怎么回答**

我项目里限流最明确，其他像超时和负载保护更多是知道概念和部分依赖框架默认。

**容易踩坑的错误回答**

不要把熔断和限流混成一个东西。

### 43. 你项目里真正用到了哪些，哪些只是知道概念？

**面试官为什么会问这个**

这题最能测真伪。

**简洁回答版**

真正明确用到的是限流、配置、日志、服务发现；熔断和负载保护我知道原理，但没有做特别重的业务化定制。

**展开回答版**

我的重点更多放在消息链路设计上，不会硬说自己把 go-zero 所有治理特性都深度用过。

**结合我项目怎么回答**

Gateway 里我确实加了 token bucket 限流；Logic 发现用了 etcd；其他治理能力更多是框架层默认和概念理解。

**容易踩坑的错误回答**

不要为了好听说“都用了”，这是最容易露馅的地方。

### 44. 如果面试官问“你用过 go-zero 的限流/熔断吗”，怎么真实回答？

**面试官为什么会问这个**

追问诚实度。

**简洁回答版**

限流我实际用了，熔断我更多是理解原理和知道框架能力，没有专门做重业务熔断策略。

**展开回答版**

项目里我在 Gateway 接口和 WebSocket 接入层用了令牌桶限流；熔断这块没有自己手写复杂规则，更多依赖服务调用超时和错误传播。

**结合我项目怎么回答**

这就是“真实但不掉价”的讲法。

**容易踩坑的错误回答**

不要说“熔断也用了”却说不出场景。

### 45. 什么是自适应负载保护（load shedding）？可以怎么浅讲？

**面试官为什么会问这个**

go-zero 常见追问。

**简洁回答版**

就是服务感觉自己快扛不住时，主动拒绝一部分请求，保证整体不被打死。

**展开回答版**

与其所有请求都排队把系统拖垮，不如有选择地拒绝一部分低优先级或超时风险高的请求。

**结合我项目怎么回答**

我项目里没有专门去调这个机制，但我理解它在高并发接入层很重要。

**容易踩坑的错误回答**

不要把它说成普通限流，限流是按规则挡流量，load shedding 更像过载保护。

### 46. 这些治理能力为什么在高并发系统里重要？

**面试官为什么会问这个**

宏观价值题。

**简洁回答版**

因为高并发下真正危险的不是某一次慢，而是连锁放大导致整体雪崩。

**展开回答版**

限流、超时、熔断、负载保护本质上都是为了控制故障传播范围。

**结合我项目怎么回答**

我项目里长连接多、消息链路长，所以接入层资源保护尤其重要。

**容易踩坑的错误回答**

不要答成“为了性能更高”，它更偏稳定性和可用性。

## 七、和项目强相关的面试追问

### 47. 你的 go-zero 服务目录结构是什么样的？

**面试官为什么会问这个**

看你是否真的看过项目。

**简洁回答版**

Gateway 走 `config/handler/logic/svc/types`，Logic 走 `config/server/logic/svc`，Transfer 是单独 worker。

**展开回答版**

这是典型的 go-zero API 与 zRPC 两套分层。

**结合我项目怎么回答**

Gateway 负责接入，Logic 负责业务，Transfer 负责 Kafka 异步消费。

**容易踩坑的错误回答**

不要答成只有一个固定模板，API 和 RPC 是不一样的。

### 48. handler / logic / svc / config 分别干什么？

**面试官为什么会问这个**

高频分层题。

**简洁回答版**

handler 接请求，logic 做业务编排，svc 放依赖，config 放配置。

**展开回答版**

handler 不该太重，logic 才是主要编排层，svc 统一注入 Redis、RPC client、Kafka writer 等。

**结合我项目怎么回答**

我项目里 Gateway 和 Logic 两边都是这么落的，只是 RPC 侧多了 `server` 入口层。

**容易踩坑的错误回答**

不要把 `svc` 说成业务层，它是依赖层。

### 49. 为什么 go-zero 倾向把业务逻辑放在 logic 层？

**面试官为什么会问这个**

设计理念题。

**简洁回答版**

因为这样 handler 可以更薄，协议处理和业务处理能分开。

**展开回答版**

这样更容易测试、更容易复用，也不容易把请求协议细节和业务规则耦合在一起。

**结合我项目怎么回答**

我项目里 Gateway handler 基本只做接入，真正调用 LogicRouter 和组织业务在 logic 层。

**容易踩坑的错误回答**

不要说“handler 里不能写任何业务”，轻业务编排可以有，但不要太重。

### 50. 这样分层有什么好处？

**面试官为什么会问这个**

泛化价值题。

**简洁回答版**

职责清楚、依赖集中、可测试性更好。

**展开回答版**

你看目录就知道哪层干什么，定位问题和改代码都会更快。

**结合我项目怎么回答**

我的 API 和 RPC 服务都因为这个分层比较好讲，也更好维护。

**容易踩坑的错误回答**

不要只答“代码规范”，太虚。

### 51. 如果 logic 写得很重，会有什么问题？

**面试官为什么会问这个**

反向思考题。

**简洁回答版**

会变成 God Object，越来越难测试、难拆分、难维护。

**展开回答版**

logic 层太重时，应该继续往更细的 domain service 或公共包拆。

**结合我项目怎么回答**

我项目里就把更核心的消息处理下沉到了 `internal/logic`，避免 `cmd/logic/internal/logic` 变成巨大文件。

**容易踩坑的错误回答**

不要说“logic 重一点没关系”，长期会失控。

### 52. 你在项目里有没有遇到 go-zero 生成代码后还要手改很多的情况？

**面试官为什么会问这个**

真实开发题。

**简洁回答版**

有，而且很正常。

**展开回答版**

脚手架解决的是骨架，不是复杂业务；像 WebSocket 长连接、Redis 在线路由、Kafka 群聊异步化、worker pool 都是我后面自己补的。

**结合我项目怎么回答**

这个项目里手写部分远比生成部分更体现业务能力。

**容易踩坑的错误回答**

不要把 goctl 神化成“全部自动化”。

### 53. 你有没有封装公共逻辑，避免 handler/logic 重复？

**面试官为什么会问这个**

看你有没有抽象能力。

**简洁回答版**

有，像 `internal/delivery`、`internal/discovery`、`internal/server` 这些就是公共能力封装。

**展开回答版**

我尽量把跨服务可复用的能力抽到外层 `internal`，而不是每个 cmd 目录都各写一份。

**结合我项目怎么回答**

消息投递、服务发现、连接管理都做了公共封装。

**容易踩坑的错误回答**

不要回答成“没有，逻辑都直接写在 handler/logic 里”。

### 54. 你的配置管理、日志、错误处理是怎么做的？

**面试官为什么会问这个**

工程能力题。

**简洁回答版**

配置用 go-zero 的 conf + yaml，再加环境变量覆盖；日志用 logx；错误尽量往上返回，由入口层统一处理。

**展开回答版**

我项目里 Gateway 和 Logic 启动时都会先读配置，并支持从环境变量覆盖关键项。

**结合我项目怎么回答**

Redis 地址、JWT secret、Gateway ID、RPC 端口这些都有环境变量覆盖逻辑。

**容易踩坑的错误回答**

不要说“错误处理全靠 panic”或者“日志随便 print”。

### 55. go-zero 在工程规范上对你帮助最大的一点是什么？

**面试官为什么会问这个**

价值总结题。

**简洁回答版**

最大帮助是让服务入口、依赖注入和分层边界很清楚。

**展开回答版**

这样我可以把精力放在消息路由、ACK、Kafka 异步链路这些业务问题上，而不是一开始就自己搭骨架。

**结合我项目怎么回答**

对我这个项目来说，这点尤其重要，因为既有 API 也有 RPC。

**容易踩坑的错误回答**

不要又回到“性能高”，还是工程化价值更稳。

## 八、结合 IM 项目的深挖

### 56. 你的 Gateway 为什么要多实例？

**面试官为什么会问这个**

IM 特有高频题。

**简洁回答版**

因为 WebSocket 长连接都挂在 Gateway 上，单实例很快会成为容量瓶颈和单点。

**展开回答版**

多实例后，连接和请求可以分散到多个 Gateway。

**结合我项目怎么回答**

我本地模拟了 10 个 Gateway，验证跨节点单聊和长连接承载。

**容易踩坑的错误回答**

不要只说“为了高并发”，单点问题也要说。

### 57. WebSocket 长连接为什么一般放在接入层？

**面试官为什么会问这个**

架构边界题。

**简洁回答版**

因为接入层最靠近客户端，更适合做握手、鉴权、心跳和连接生命周期管理。

**展开回答版**

核心业务服务更适合保持相对无连接化，不应该绑死大量连接状态。

**结合我项目怎么回答**

我的 Gateway 就承担了 JWT 校验、心跳、在线状态刷新和连接管理。

**容易踩坑的错误回答**

不要答成“只是习惯这么放”，要说出原因。

### 58. 为什么消息服务不直接绑死在某一个 Gateway 上？

**面试官为什么会问这个**

关键分布式设计题。

**简洁回答版**

因为用户会连到不同 Gateway，如果业务绑死在某一个节点上，跨节点路由和扩容都会变麻烦。

**展开回答版**

接入层和业务层解耦后，任意 Gateway 都能收消息，再统一转给 Logic 处理。

**结合我项目怎么回答**

这就是我为什么拆 Gateway 和 Logic。

**容易踩坑的错误回答**

不要说“绑死也可以，只是不好”，要说出它会限制扩展性。

### 59. 单聊和群聊链路在 go-zero 架构下怎么讲？

**面试官为什么会问这个**

项目核心题。

**简洁回答版**

单聊是 Gateway 收消息 -> RPC 调 Logic -> Logic 查在线状态并走 RedisDelivery；群聊是 Gateway -> Logic -> Kafka job -> Transfer 消费 -> RedisDelivery。

**展开回答版**

两者前半段都经过 Gateway/API 和 Logic/RPC，差别在于群聊把扇出拆到 Kafka + Transfer。

**结合我项目怎么回答**

这条回答最贴你项目。

**容易踩坑的错误回答**

不要把群聊也说成直接在 Logic 里同步 for 循环扇出。

### 60. 群聊为什么要异步扩散？和 go-zero 架构怎么配合？

**面试官为什么会问这个**

第四天核心题。

**简洁回答版**

因为群聊成员多，同步扇出会拖慢 Logic 主链路，所以先写 Kafka，再由 Transfer 后台扩散。

**展开回答版**

go-zero 负责把 Gateway 和 Logic 的职责边界搭清楚，Kafka/Transfer 则承接重扇出链路。

**结合我项目怎么回答**

Logic 里把群聊抽象成 `groupDispatchJob`，Transfer 异步消费后逐个 `Deliver`。

**容易踩坑的错误回答**

不要把 Kafka 说成“读写分离”，它在这里是异步解耦。

### 61. Redis、Kafka、MySQL 在你的 go-zero 项目里各起什么作用？

**面试官为什么会问这个**

综合中间件题。

**简洁回答版**

Redis 管在线状态、ACK、离线补偿和 Pub/Sub；Kafka 管群聊异步扩散、重试、死信；MySQL 管用户和历史消息。

**展开回答版**

三者刚好对应状态中心、异步链路和持久化。

**结合我项目怎么回答**

这是我项目里最清晰的一组分工。

**容易踩坑的错误回答**

不要说 Redis 只是缓存、Kafka 只是队列，这样太浅。

### 62. 你的 go-zero 架构里，哪一层最容易成为瓶颈？

**面试官为什么会问这个**

容量与瓶颈判断题。

**简洁回答版**

接入层 Gateway 和 Redis 最容易先成为瓶颈，具体看是连接压力大还是消息扇出大。

**展开回答版**

长连接数量大时 Gateway 压力先上来，跨节点投递和在线状态查询多时 Redis 压力也明显；群聊多时 Kafka/Transfer 也可能成为瓶颈。

**结合我项目怎么回答**

从我现在的验证看，Gateway 接入层是最先被关注的一层。

**容易踩坑的错误回答**

不要机械回答“数据库一定是瓶颈”，要结合 IM 场景。

### 63. 如果用户量再扩大 10 倍，你优先扩哪一层？

**面试官为什么会问这个**

演进题。

**简洁回答版**

先扩 Gateway，然后看 Redis 和 Logic，再看 Kafka/Transfer。

**展开回答版**

因为 IM 最直接的压力通常先打在长连接接入层，其次是在线状态中心和消息编排层。

**结合我项目怎么回答**

如果单聊占比高，我会优先扩 Gateway 和 Redis；如果群聊扇出很多，再重点扩 Transfer 和 Kafka 消费能力。

**容易踩坑的错误回答**

不要一上来就答“扩数据库”。

### 64. 如果让你把这个项目继续往“更完整的微服务”演进，你会怎么拆？

**面试官为什么会问这个**

架构演进题。

**简洁回答版**

我会优先把在线状态/路由、群组服务、消息存储和补偿能力进一步独立出来。

**展开回答版**

比如 Presence/Route 服务、Group Service、Message Storage Service、Delivery/Sync Service，而不是让 Logic 一直变大。

**结合我项目怎么回答**

但我会强调当前版本还没必要一开始拆太细，先把主要链路跑稳。

**容易踩坑的错误回答**

不要一口气拆十几个服务，没有边界依据会显得空。

## 九、容易被问住的概念辨析

### 65. 多实例部署是不是单体？

**面试官为什么会问这个**

辨析题。

**简洁回答版**

代码层面可能还是单仓库/单应用族，但部署层面已经不是单机单体了。

**展开回答版**

多实例部署更像“分布式部署的同类服务”，不等于完整微服务，但也不只是传统单机单体。

**结合我项目怎么回答**

我项目就是这种典型中间态。

**容易踩坑的错误回答**

不要非黑即白。

### 66. 分布式和集群有什么区别？

**面试官为什么会问这个**

高频基础题。

**简洁回答版**

集群通常强调同一种服务多实例协同，分布式更强调多个节点、多个组件之间协作完成一个系统。

**展开回答版**

集群是分布式的一种常见形态，但分布式范围更广。

**结合我项目怎么回答**

我这里 Gateway 多实例是集群特征，Gateway/Logic/Transfer 协作是分布式特征。

**容易踩坑的错误回答**

不要答成完全没区别。

### 67. 微服务和分布式是不是一个意思？

**面试官为什么会问这个**

概念边界题。

**简洁回答版**

不是，微服务是分布式系统的一种拆分方式。

**展开回答版**

你可以有分布式部署但不是细粒度微服务，也可以说微服务一定是分布式的一种实现。

**结合我项目怎么回答**

我项目是分布式协作明显，但还不是完全拆到底的微服务体系。

**容易踩坑的错误回答**

不要把两者直接等同。

### 68. API 网关 和 你项目里的 Gateway 是一回事吗？

**面试官为什么会问这个**

很容易混。

**简洁回答版**

不完全一样，我的 Gateway 更像接入层节点，兼具一些 API 网关职责，但不是那种统一协议聚合和流量编排平台。

**展开回答版**

API Gateway 这个词在大厂里常指统一入口平台，而我这里的 Gateway 更强调 WebSocket 接入、鉴权、限流和消息转发。

**结合我项目怎么回答**

它既接 `/login`、`/ws`，又做连接管理。

**容易踩坑的错误回答**

不要直接说“一回事”，也不要说“完全没关系”。

### 69. 服务注册发现 和 负载均衡 的关系是什么？

**面试官为什么会问这个**

高频基础题。

**简洁回答版**

注册发现先告诉你有哪些节点，负载均衡再从这些节点里选一个。

**展开回答版**

没有发现，你连候选列表都没有；没有负载均衡，你有列表也不知道该选谁。

**结合我项目怎么回答**

我项目里 Etcd 负责发现，zRPC `p2c_ewma` 负责选择 Logic。

**容易踩坑的错误回答**

不要把两者混成一个概念。

### 70. RPC 调用 和 MQ 异步调用 的区别是什么？

**面试官为什么会问这个**

同步异步边界题。

**简洁回答版**

RPC 是同步请求-响应，MQ 是异步投递-消费。

**展开回答版**

RPC 适合需要立即结果的调用，MQ 适合削峰和解耦。

**结合我项目怎么回答**

Gateway 调 Logic 用 RPC，群聊扩散用 Kafka。

**容易踩坑的错误回答**

不要说“MQ 性能更高所以都用 MQ”，要看业务语义。

### 71. 一致性哈希是解决什么问题的？

**面试官为什么会问这个**

你已经被问过很多次了。

**简洁回答版**

解决“从一组可用节点里，稳定地给同一个 key 选一台节点”的问题。

**展开回答版**

它不解决服务发现，只解决稳定选路。

**结合我项目怎么回答**

我这里用于 Gateway 按用户维度选 Logic。

**容易踩坑的错误回答**

不要说它是找用户在哪台 Gateway，那是 Redis route。

### 72. go-zero 只是代码生成工具吗？

**面试官为什么会问这个**

典型质疑题。

**简洁回答版**

不是，它既有代码生成，也有 API/RPC 运行时和治理能力。

**展开回答版**

goctl 只是生成骨架，go-zero 本身还包括 API server、zRPC、配置、服务发现等。

**结合我项目怎么回答**

我项目里服务启动、ServiceContext、RPC 边界都不是单纯“生成代码”能概括的。

**容易踩坑的错误回答**

不要把 go-zero 简化成脚手架。

### 73. go-zero 和 Spring Cloud 在思路上有什么相似点和区别？

**面试官为什么会问这个**

跨语言对比题。

**简洁回答版**

相似点是都强调服务化、注册发现、治理和工程规范；区别是 go-zero 更轻量，Go 风格更直接，生态没有 Spring Cloud 那么“大而全”。

**展开回答版**

你不用展开一堆组件对照，知道它们都是“服务治理型框架体系”就够了。

**结合我项目怎么回答**

我这里最像 Spring Cloud 的点，就是 API/RPC 拆分、服务发现和工程化分层。

**容易踩坑的错误回答**

不要硬比生态厚度，Go 和 Java 场景不一样。

## B. 5 段 1 分钟口语回答模板

### 1. 自我介绍式项目总述

我做的是一个基于 Go 的 IM/长连接后端项目，整体上拆成 Gateway、Logic 和 Transfer 三层。Gateway 用 go-zero API 承接 HTTP 和 WebSocket 接入，主要负责鉴权、限流、心跳和连接管理；Logic 用 go-zero zRPC 负责消息编排、顺序控制、路由和持久化；Transfer 负责 Kafka 群聊异步扩散。项目里我重点做的是跨节点单聊路由、ACK 与离线补偿、群聊异步化以及多 Gateway 的长连接承载验证。

### 2. go-zero 在项目里的价值

go-zero 在我这个项目里的价值，不是替我做业务，而是把 API、RPC、依赖注入和服务启动这些基础盘先搭好。这样 Gateway 和 Logic 的职责边界非常清楚，我可以把更多精力放在消息路由、Redis 在线状态、ACK 重放和 Kafka 群聊扩散这些业务复杂点上。简单说，框架帮我省骨架，我自己解决链路问题。

### 3. Etcd + p2c_ewma 怎么讲

我项目里 Gateway 调 Logic 不是写死地址，而是先通过 Etcd 获取当前可用的 Logic 节点列表，再由 go-zero zRPC 的 `p2c_ewma` 在候选节点中做负载选择。Etcd 解决“有哪些节点在线”，p2c_ewma 解决“优先调用哪个健康节点”。Logic 是无状态业务层，因此当前不依赖用户粘滞。

### 4. 跨节点单聊路由怎么讲

跨节点单聊时，Gateway 收到用户消息后会通过 RPC 交给 Logic，Logic 先生成 `session_id`、`seq` 和 `message_id`，再查 Redis 里的 `route:<uid>`，看目标用户当前连在哪台 Gateway。查到之后，Logic 会把消息发布到目标 Gateway 对应的 Redis Pub/Sub 频道，由那台 Gateway 再通过本机 WebSocket 连接推给用户。如果用户不在线，就走离线补偿。

### 5. 群聊为什么要 Kafka

群聊和单聊最大的区别不是协议，而是扇出规模。单聊只发给一个人，Logic 可以同步直投；群聊可能要发给很多成员，如果 Logic 直接同步 for 循环扇出，会把主链路拖慢。所以我把群聊抽象成一张 group dispatch job 写进 Kafka，再由 Transfer 后台异步消费，逐个复用单聊那套 RedisDelivery 去投递。这样既做了解耦，也做了削峰，还能挂 retry 和 dead-letter。

## C. 一条典型连续追问链

### Q1：go-zero 是什么？

它是一套 Go 服务端工程化框架，不只是 HTTP，还包括 API、RPC、服务发现和一些治理能力。

### 追问 1：那你项目里哪里体现了 go-zero？

Gateway 用 API，Logic 用 zRPC，`handler/logic/svc` 和 `server/logic/svc` 分层是最直观的体现。

### 追问 2：那你为什么还要做 RPC 拆分？

因为 Gateway 是接入层，Logic 是业务层，长连接管理和消息编排关注点不同，拆开更利于扩展。

### 追问 3：Gateway 怎么找到 Logic？

Gateway 先通过 Etcd 获取可用 Logic 节点，再通过 zRPC `p2c_ewma` 选择节点。

### 追问 4：p2c_ewma 解决什么？

它不解决服务发现，而是在已发现的节点中结合 EWMA 负载估计做选择，减少请求打到慢节点的概率。

### 追问 5：那消息最终怎么到另一个 Gateway？

Logic 通过 Redis 的 `route:<uid>` 找目标用户所在 Gateway，再按 `im_message_push:<gatewayID>` 定向 Pub/Sub 投递。

### 追问 6：如果用户不在线呢？

就把消息写到离线补偿集合，同时保留未确认消息索引，用户重连后再按顺序重放。

### 追问 7：群聊为什么不用这套同步逻辑直接发？

因为群聊扇出规模大，会阻塞 Logic 主链路，所以改成 Kafka + Transfer 异步扩散。

### 追问 8：那你这个项目里最体现你能力的不是框架，而是什么？

不是 go-zero 本身，而是我基于它做出来的跨节点路由、ACK/离线补偿和群聊异步链路设计。
