# AI 原生后端岗位匹配说明

## 1. 岗位要求怎么拆

这个岗位不是纯算法岗，更像：

```text
后端工程能力
  +
AI 应用工程能力
  +
Agent / RAG / 工具调用落地经验
```

所以不能只靠一个 IM 项目，也不能只靠一个 Agent demo。最合适的简历组合是：

```text
项目一：LinkGo Chat：AI 好友与红包协同 IM 系统
项目二：基于 DeerFlow 的项目级 AI 代码变更任务流平台
```

## 2. 项目一能证明什么

LinkGo Chat 主要证明工程底子：

```text
1. Go / go-zero / gRPC / WebSocket 工程能力
2. Redis 在线路由、ACK、离线补偿和消息幂等
3. Kafka 群聊异步扩散
4. MySQL 事务、行锁、唯一索引和红包并发一致性
5. AI 好友进入 IM 消息链路，不是单独外挂接口
6. AI 调用审计、provider fallback 和演示闭环
```

它对应岗位里的：

```text
编程底子厚
代码编写与工程实践能力
AI 能力结合真实业务落地
快速迭代与业务结果意识
```

项目一不要硬塞 MCP、多 Agent 和复杂 RAG。这个项目的定位是：

```text
Go 后端主项目 + AI 聊天好友业务增强
```

## 3. 项目二能证明什么

DeerFlow 二开项目主要证明 AI 工程能力：

```text
1. 接手开源 Agent 框架并做二次开发
2. 项目级上下文管理
3. 代码仓库索引和 RAG
4. 任务编排、沙箱执行、测试回归
5. PR / Diff / 审计闭环
6. 多步骤 Agent 工作流理解
```

它对应岗位里的：

```text
LLM / Agent / RAG 原理
AI 编程工具和 AI 原生开发方式
多智能体、Function Calling、工具调用类项目经验
```

如果岗位强问 MCP，可以这样讲：

```text
我没有在项目一里强塞 MCP，因为 IM 项目主线是实时通信。
MCP / Tool Calling 更适合放在项目二的 Agent 工具执行层：Agent 需要调用仓库检索、测试执行、Git 操作和 PR 创建，这些工具边界更清晰。
```

## 4. 当前匹配度判断

结论：

```text
两个项目组合是匹配这个岗位方向的。
单看项目一，只能算“Go 后端 + AI 应用增强”。
加上项目二，才能完整覆盖 Agent / RAG / 工具调用 / AI 工程闭环。
```

当前还需要补强的点：

```text
1. 项目一：继续完善 AI 好友私聊的真实模型接入、超时降级和演示脚本。
2. 项目一：补红包并发测试证据和面试解释材料。
3. 项目二：把 DeerFlow 二开的项目级 workflow、代码索引、沙箱测试和 PR 闭环讲清楚。
4. 简历：不要写“会 Cursor / Claude Code”，要写“使用 AI 工具完成需求拆解、代码修改、测试回归和文档沉淀”。
```

## 5. 简历命名建议

项目一推荐写：

```text
LinkGo Chat：AI 好友与红包协同 IM 系统
```

项目二推荐写：

```text
DeerFlow 二开：项目级 AI 代码变更任务流平台
```

这两个名字组合起来，比“企业 IM + AI 平台”更自然，也更贴近岗位。
