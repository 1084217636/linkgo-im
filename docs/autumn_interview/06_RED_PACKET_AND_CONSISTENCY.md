# 06 红包与并发一致性

## 定位

红包不是支付系统，而是用来证明事务、行锁、唯一索引和幂等能力的并发业务。

## 创建链路

```text
校验发送者/会话/金额/个数
-> MySQL 事务写 red_packets
-> 返回 red_packet_id
```

金额使用整数“分”，避免浮点误差。创建时要求总金额、个数为正，并设置过期时间和初始状态。

## 领取链路

```text
开始事务
-> SELECT 红包 FOR UPDATE
-> 检查状态、过期、剩余数量/金额
-> 检查用户是否已领取
-> 计算本次金额
-> UPDATE 剩余金额/数量
-> INSERT red_packet_claims
-> 必要时标记 finished
-> COMMIT
```

## 为什么需要行锁

多个用户同时领取时都在修改同一红包剩余量。`FOR UPDATE` 让同一红包的扣减串行进入临界区，避免读到相同剩余值后重复扣减。

## 为什么还要唯一索引

应用层先查再写存在并发窗口。`red_packet_id + user_id` 唯一索引是数据库最终防线。同一个用户重复请求只产生一条领取记录。

## 随机金额基本约束

无论采用何种随机算法，都必须保证：本次金额大于零；扣减后剩余金额足够保证其他未领取者至少拿到最小单位；最后一人拿走剩余金额；总和不超过总金额。

## 事务失败

扣减成功但明细插入失败时，整个事务回滚，不能只回滚一张表。客户端超时重试时通过领取唯一索引返回既有结果。

## 热点问题

同一个大红包的行锁可能成为热点。可演进为 Redis Lua 预扣、分段红包或异步结算，但复杂度和一致性成本更高。校招项目当前 MySQL 事务方案更容易证明正确性。

## 常见追问

- 会超卖吗？事务行锁保护剩余量，唯一索引防重复领取。
- 为什么金额不用 float？浮点表示误差会破坏金额精确性。
- 数据库宕机怎么办？事务未提交则回滚；提交成功但响应丢失则幂等查询既有结果。
- 是否是真实支付？不是，没有资金账户、清结算、风控和合规。
