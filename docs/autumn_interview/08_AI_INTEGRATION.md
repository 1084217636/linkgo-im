# 08 AI 能力与正确边界

## 定位

AI 是 IM 上层增强，不是项目主链路。核心价值仍是 Go 长连接、可靠投递、并发一致性和运营平台。

## 已实现能力

- 群聊总结：读取有权限的群历史，生成摘要、待办和风险。
- 知识问答：从项目 FAQ/知识文档召回相关片段，再调用 provider 回答。
- Provider 抽象：mock、OpenAI-compatible/DeepSeek 等。
- timeout、fallback、调用审计、attempt 记录、延迟指标和敏感信息脱敏。

## 群聊总结链路

```text
JWT 用户请求
-> 校验群成员权限
-> MySQL 读取限定条数历史消息
-> 构造结构化 Prompt
-> Provider 调用（带 timeout）
-> 解析 summary/todos/risks
-> 保存结果和调用审计
-> 返回客户端
```

为什么要权限校验：AI 接口不能绕过普通历史消息权限读取别人的群聊。

## 知识问答

当前是轻量 FAQ/RAG：加载项目文档，按关键词/相关性取 TopK，连同问题交给模型，返回答案和 sources。

正确表述：最小可解释检索闭环。不能说成大规模向量数据库或复杂训练平台。

## Provider 抽象

业务服务依赖统一接口，不直接绑定厂商 SDK。切换模型主要通过配置，mock 用于稳定本地测试。

好处：可测试、可替换、可做 fallback；代价：不同模型能力和参数不能完全被一个接口抹平。

## 超时与降级

外部模型可能慢、限流或失败。调用必须设置 timeout；主 provider 失败可根据策略降级到 mock/备用 provider，同时记录每次尝试和最终结果。

AI 不放入实时消息强依赖：模型失败不能导致普通聊天不可用。

## 安全

- API Key 通过环境变量/Secret，不提交 Git。
- 日志对 Token、Key 和敏感错误脱敏。
- 限制历史条数和输入长度。
- 结果是模型生成内容，不能作为绝对事实。
- 防止 Prompt 中的用户内容被误当系统权限指令。

## 面试边界

不能说：训练了模型、完整向量 RAG、Agent 自主操作生产、AI 结果完全正确。

可以说：实现 provider 工程接入、检索上下文、权限、超时、fallback、审计和指标，并把 AI 与实时主链路隔离。
