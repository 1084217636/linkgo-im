# 教学型总结

这份总结默认你一开始对 Etcd、Kafka、Protobuf、ACK、限流这些概念都不熟。目标不是背名词，而是理解“为什么项目会一步步升级成现在这样”。

## 1. 先理解原始问题

一个最基础的 IM 项目通常只要做到三件事：

- 客户端能连上 WebSocket
- 用户能互相发消息
- 消息能存数据库

但这只能叫“能聊天”，还不叫“分布式 IM”。因为一旦服务拆成多节点，就会出现很多真实问题：

- 用户连在 A 网关，消息怎么准确发到他？
- 同一个会话里的消息怎么保证顺序？
- 客户端断线了，哪些消息没真正收到？
- 群聊很大时，为什么不能每次都同步一条条推？
- 服务节点变多以后，网关怎么知道该连哪台 Logic？
- 系统出问题时，怎么知道是 ACK 异常、Kafka 堆积，还是限流打满了？

所以升级版项目，其实就是围绕这些问题逐个补能力。

## 2. 为什么先做消息协议升级

如果消息一开始只是随意 JSON，短期开发快，但后面会越来越乱：

- 字段名不统一
- 心跳、普通消息、ACK 很难整齐区分
- 字段扩展靠约定，容易失控

所以改成 Protobuf 的本质，不是“为了高级”，而是“为了让消息长得统一”。统一之后，系统里每一层都更容易沟通。

你可以把 Protobuf 理解成：
先设计一张标准表格，再把它压缩成二进制快递箱寄出去。

## 3. 为什么要 Etcd

以前 Gateway 如果直接写死 `logic:9001`，那只是单节点写法。一旦 Logic 变成多实例，Gateway 需要知道：

- 现在有几台 Logic
- 哪台是活的
- 某个用户的请求尽量该去哪台

Etcd 就像“服务通讯录”。Logic 启动时往里面登记，Gateway 再去查。

为什么还要加一致性哈希？
因为如果每次都随机选 Logic，同一个用户的请求可能总在不同节点间跳来跳去，不稳定。用哈希选节点，等于“尽量让同一个用户总去相对固定的节点”，这样路由更稳。

## 4. 为什么 Redis 不只是缓存

很多初学者只把 Redis 当缓存，其实在 IM 里 Redis 经常承担更多角色：

- `route:<uid>`：记录用户当前在线并且在哪个网关
- `Pub/Sub`：跨网关实时推送消息
- `pending_ack`：记录客户端还没确认的消息
- `offline_msg`：记录离线期间要补发的消息
- `group_members`：存群成员

所以这里的 Redis 更像“实时状态中心”。

## 5. 为什么要 Lua Sequence

如果两个用户同时发消息，你只用时间戳，很难绝对保证一个会话内顺序稳定。

所以更可靠的做法是：
给每个会话维护一个递增序号 `seq`。

每来一条消息就加一，下一条一定比上一条大。

为什么用 Lua？
因为 Lua 脚本能在 Redis 里原子执行，“读旧值 + 加一 + 写回”是一整步，不会被并发打断。

## 6. 为什么 ACK 很重要

很多人误以为“消息从服务端发出去了”就等于“消息到了”。其实不是。

真实情况是：

- 服务端发了
- 网络抖了一下
- 客户端没处理
- 服务端如果没有 ACK，就不知道这条消息到底成功没有

所以 ACK 的意义是：
客户端收到并确认后，服务端才能把这条消息从待确认集合里删掉。

这就是为什么项目里有 `pending_ack` 和 `ack_idx`。

## 7. 为什么群聊要上 Kafka

单聊只发给一个人，Logic 直接处理没问题。

群聊不一样。一个群如果 500 人，Logic 如果同步循环投递 500 次，会让主链路明显变慢。

Kafka 的思路是：

- Logic 不直接扩散
- Logic 先把“这条群消息要发给哪些人”写到 Kafka
- Transfer 再慢慢消费并扩散

这样主链路就不会被群聊扇出拖死。

Kafka 本质上是一个“高吞吐排队系统”。

## 8. 为什么还要 retry 和 dead-letter

很多项目做到 Kafka 就停了，但这还不够。

因为消费也会失败，比如：

- Redis 写失败
- 数据格式异常
- 某一步序列化失败

所以要分层处理：

- 先重试几次
- 还不行，就进死信队列

死信不是“丢掉”，而是“先隔离起来，方便排查和补偿”。

## 9. 为什么加 worker pool

如果每条消息来都直接 `go func()`，在流量大时 goroutine 可能飙升。

worker pool 的作用就是：

- 先把任务丢到队列
- 固定数量 worker 去处理
- 把系统资源上限控制住

你可以把它理解成：
不是来一个任务就临时招一个新工人，而是固定一批工人轮流干活。

## 10. 为什么要 Prometheus 指标

如果没有指标，线上出问题时你只能看日志，很被动。

Prometheus 指标的意义是让你直接看到：

- 当前多少连接
- 消息有没有大量解码失败
- ACK 是否回收正常
- Kafka 是否在重试
- 死信是不是在增加
- 限流是不是被频繁触发

所以指标系统不直接创造业务价值，但它能让你有“系统正在发生什么”的感知能力。

## 11. 这次升级的真正思路

如果用一句话概括这次改造，不是“加了一堆中间件”，而是：

把一个基础聊天项目，升级成了一个能解释分布式路由、消息可靠性、异步削峰、资源保护和可观测性的工程化 IM 项目。

## 12. 你现在最该掌握的表达方式

以后你讲这个项目，不要背技术栈列表，要按这个顺序讲：

1. 原始问题是什么
2. 为什么原方案不够
3. 为了解决这个问题引入了什么技术
4. 技术引入后链路怎么变化
5. 这样做带来了什么收益

只要你能按这个顺序讲，面试官会感觉你不是“背项目”，而是真懂这个项目为什么这么设计。
