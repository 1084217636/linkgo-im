## 📂 项目结构全解析与简历对应图

这是你目前项目的逻辑架构，它体现了**大厂经典的“网关层+逻辑层”分离架构**。

### 1. `cmd/` (服务入口层)

这里存放的是两个可以独立启动的服务，体现了简历中的“解耦业务逻辑与长连接管理”。

* **`cmd/gateway/main.go` (网关层/接入层)**
* **职责**：负责维持成千上万个用户的 WebSocket 长连接。它像是一个“前台”，只负责把用户接进来。
* **对应简历**：**“基于 Gin 框架快速构建...集成 WebSocket 协议实现双工通信”**。
* **核心代码点**：
* `upgrader.Upgrade(...)`：将 HTTP 协议升级为 WebSocket 协议。
* `userConns sync.Map`：这就是你简历里写的**“连接池管理”**，用来在内存里存住在设备上在线的用户。
* `ws.SetReadDeadline(...)`：这是**“心跳检测机制”**的后端实现，配合前端发的 PING，如果 60 秒没收到动静就自动断开，释放服务器资源。




* **`cmd/logic/main.go` (业务逻辑层)**
* **职责**：它是系统的“大脑”，不直接跟用户连，只通过 gRPC 听网关的指令。它决定了消息是该存到 Redis 还是该发给某个网关。
* **对应简历**：**“实现具备高并发处理能力的即时通信后端”**。



---

### 2. `internal/` (核心业务实现)

这是你项目的“肌肉”，包含了简历里最值钱的技术点。

* **`internal/logic/handler.go` (消息分发逻辑)**
* **核心逻辑**：这是你实现**“在线直发+离线补偿”**的地方。
* **对应简历**：**“利用 Redis 的发布/订阅 (Pub/Sub) 模式实现多节点间的消息广播”**。
* **面试必杀技**：如果面试官问“用户在 A 网关，目标在 B 网关，怎么发消息？”，你就说：“我用 `rdb.Publish` 把消息丢进 Redis 频道，所有网关都订阅了这个频道，哪个网关发现目标在自己这，谁就推送。”。
* **离线逻辑**：`h.Rdb.ZAdd(ctx, key, redis.Z{Score: ..., Member: ...})`。这就是你简历里的**“基于 Redis ZSet 的消息暂存逻辑”**。


* **`internal/middleware/auth.go` (鉴权中间件)**
* **核心逻辑**：检查用户连接时带的 Token 是否合法。
* **对应简历**：**“封装鉴权中间件以保障连接安全性”**。
